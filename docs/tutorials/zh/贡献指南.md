## 开发限制和指南

为了保持项目代码质量和一致性，请遵守以下指南。

### 语言要求

- **所有代码应使用英文编写**：包括变量名、函数名、注释、docstring、提交信息和文档。
- 我们理解贡献者可能来自不同背景，但使用英文可以让代码库对更广泛的用户保持可读和可维护。

### 模块化和可复用性

- **编写模块化、可复用的代码**：函数和脚本应尽量具有灵活性。
- **优先通过参数传递信息，避免硬编码**：使用函数参数、命令行参数或配置文件，而不是把数值直接写死。不过，某些文件名是 GPUMD 和 NEP 程序固定要求的，例如 `train.xyz`、`thermo.out`，这类情况可以按需要保留固定文件名。
- **示例**：

  ```bash
  # Good: accepts parameters
  function convert_file() {
      local input_file=$1
      local output_file=$2
      # processing logic
  }

  # Avoid: hardcoded values (unless required by GPUMD/NEP)
  function convert_file() {
      local input_file="hardcoded_input.xyz"
      # processing logic
  }
  ```

### 代码风格

- 编写清晰、易于他人维护和修改的代码。
- 使用有意义的变量名和函数名。
- 在必要时添加注释，解释复杂逻辑。

### 文档

- 确保帮助信息（例如 `-h` 参数）清晰、准确。
- `docs/` 目录中的文档通常由维护者更新，因此一般不需要贡献者直接修改。

## 贡献代码

### Fork 并克隆仓库

1. 在 GitHub 的 [GPUMDkit 仓库页面](https://github.com/zhyan0603/GPUMDkit)右上角点击 "Fork"，将仓库 fork 到自己的账号下。

2. 将你的 fork 克隆到本地：

   ```bash
   git clone https://github.com/YOUR_USERNAME/GPUMDkit.git
   cd GPUMDkit
   ```

3. 设置 upstream remote，方便和主仓库保持同步：

   ```bash
   git remote add upstream https://github.com/zhyan0603/GPUMDkit.git
   ```

### 创建功能分支

为你的修改创建一个新分支：

```bash
# Create and checkout a new branch from main
git checkout -b your-branch-name
```

分支名可以按你的习惯命名。

### 进行修改

`GPUMDkit` 的结构包括：

- **`gpumdkit.sh`**：主入口 Bash 脚本，负责交互菜单模式和命令行模式。
- **`Scripts/`**：按功能分类组织的 Python 工具脚本。
  - `plt_scripts/`：绘图脚本
  - `calculators/`：计算工具
  - `format_conversion/`：格式转换工具
  - `workflow/`：工作流自动化
  - `sample_structures/`：结构采样工具
  - `analyzer/`：分析工具
  - `utils/`：实用函数，包括用于 Tab 补全的 `completion.sh`
- **`src/`**：包含交互模式菜单函数的 Shell 脚本。
  - `f1_format_conversions.sh`
  - `f2_sample_structures.sh`
  - `f3_workflows.sh`
  - `f4_calculators.sh`
  - `f5_analyzers.sh`
  - `f6_plots.sh`
  - `f7_utilities.sh`
- **`docs/`**：文档文件。

#### 交互模式贡献

如果要添加一个可通过交互菜单访问的新功能：

1. **在合适的 `Scripts/` 子目录中创建实现脚本**：

   ```bash
   # Example: Add a new format conversion script
   touch Scripts/format_conversion/new_converter.py
   ```

2. **实现你的功能**，并遵循模块化指南：接受参数，避免不必要的硬编码。

3. **在对应的 `src/` 文件中添加包装函数**：

   ```bash
   # Example: Edit src/f1_format_conversions.sh
   vim src/f1_format_conversions.sh
   ```

   添加一个新函数：

   ```bash
   function f106_new_converter(){
   echo " >-------------------------------------------------<"
   echo " | Calling the script in Scripts/format_conversion |"
   echo " | Script: new_converter.py                        |"
   echo " | Developer: Your Name (your@email.com)           |"
   echo " >-------------------------------------------------<"
   echo " Input the required parameters:"
   echo " Example: input.xyz output.lmp"
   echo " ------------>>"
   read -p " " converter_params
   echo " ---------------------------------------------------"
   python ${GPUMDkit_path}/Scripts/format_conversion/new_converter.py ${converter_params}
   echo " Code path: ${GPUMDkit_path}/Scripts/format_conversion/new_converter.py"
   echo " ---------------------------------------------------"
   }
   ```

4. **更新 `gpumdkit.sh` 中的菜单显示**：

   ```bash

   ```

5. **在 `gpumdkit.sh` 中添加 case 分支**：


#### 更新 Tab 补全

添加新的命令行 flag 时，请更新 Tab 补全脚本：

```bash

vim Scripts/utils/completion.sh
```

将新的 flag 加入 `opts` 变量：

```bash
# Find the line with local opts=...
local opts="-h -update -U -help -clean -time -plt -calc -analyze_bonds -range -out2xyz ..."

```

如果这个 flag 接受二级选项，例如 `-plt` 或 `-calc`，也需要为它添加对应的 `case` 分支：

```bash
case "$prev" in
    # ... existing cases ...
    -analyze_bonds)
        COMPREPLY=($(compgen -f -- "$cur"))  # Complete with filenames
```

### 代码风格和最佳实践

- **Shell 脚本**：
  - 使用 `${variable}` 进行变量展开。
  - 编写清晰、易维护的代码。

- **Python 脚本**：
  - 编写清晰、可维护的代码。
  - 使用有意义的变量名。

### 测试你的修改

提交贡献前，请进行测试：

1. **测试交互模式**（如果适用）：

   ```bash
   gpumdkit.sh
   # Navigate to your new feature and test it thoroughly
   ```

2. **测试命令行模式**（如果适用）：

   ```bash
   gpumdkit.sh -your_new_flag [arguments]
   gpumdkit.sh -your_new_flag -h  # Test help message
   ```

3. **测试 Tab 补全**（如果修改了 `completion.sh`）：

   ```bash
   source Scripts/utils/completion.sh
   gpumdkit.sh -<TAB>  # Should show your new flag
   ```

4. **使用不同输入测试**：

   - 测试常规输入。
   - 测试边界情况，例如空文件、大文件等。
   - 测试错误处理，例如缺少文件、参数无效等。

5. **确认没有回归问题**：确保已有功能仍然可以正常工作。

### 提交信息

提交信息应清楚、具体，说明你修改了什么。不要求严格格式，但需要让他人能够理解。

### 推送并打开 Pull Request

1. **提交你的修改**：

   ```bash
   git add .
   git commit -m "your descriptive message"
   ```

2. **让你的分支与 upstream `main` 分支保持同步**：

   ```bash
   git fetch upstream
   git rebase upstream/main
   ```

3. **将分支推送到你的 fork**：

   ```bash
   git push origin your-branch-name
   ```

4. **打开 Pull Request**：

   - 进入 [GPUMDkit repository](https://github.com/zhyan0603/GPUMDkit)。
   - 点击 "Pull Requests" -> "New Pull Request"。
   - 将 base branch 设置为 `dev`，用于代码审查。
   - 将 compare branch 设置为你的功能分支。
   - 填写 PR 描述，包括：
     - **Title**：简要描述修改内容。
     - **Description**：详细说明修改了什么以及为什么修改。
     - **Testing**：说明你如何测试这些修改。

5. **回应审查意见**：

   - 接受建议和建设性意见。
   - 及时完成要求的修改。
   - 将额外提交推送到你的分支，它们会自动显示在 PR 中。

6. **审查通过后**：维护者会合并你的 PR。

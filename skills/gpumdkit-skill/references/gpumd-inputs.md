## Reference scope

This file contains the portable input-file specification needed for normal GPUMD setup. Pair it with the setup, ensemble, compute, and output catalogs according to the keywords used by the task.

## `model.xyz`

GPUMD reads extended XYZ.

Line 1:

```text
<number_of_atoms>
```

Line 2 contains case-insensitive `keyword=value` pairs. GPUMD recognizes:

| Header field | Requirement | Meaning |
|---|---|---|
| `lattice="ax ay az bx by bz cx cy cz"` | Mandatory | Cell vectors in Angstrom |
| `pbc="T T T"` | Optional; default `T T T` | Periodicity along cell vectors |
| `properties=...` | Mandatory in practice | Per-atom column schema |

Recognized property entries:

| Property | Requirement | Unit/meaning |
|---|---|---|
| `species:S:1` | Mandatory | Case-sensitive chemical symbol |
| `pos:R:3` | Mandatory | Cartesian position, Angstrom |
| `mass:R:1` | Optional | amu; GPUMD default mass if absent |
| `charge:R:1` | Optional | elementary charge; default zero |
| `vel:R:3` | Optional | Angstrom/fs |
| `group:I:N` | Optional | `N` integer grouping methods |

Every atom row must contain exactly the columns declared by `properties`. The number of atom rows must equal line 1.

Example with one grouping method:

```text
2
lattice="5 0 0 0 5 0 0 0 5" pbc="T T T" properties=species:S:1:pos:R:3:group:I:1
Li 0.0 0.0 0.0 0
O  2.5 2.5 2.5 1
```

Do not copy this cell, species, or grouping into a real calculation. Build them from the user's structure and intended commands.

## Grouping

`group:I:N` appends `N` integer labels to each atom row. Grouping method indices run from `0` to `N-1`. A group-aware command normally takes both `<grouping_method>` and `<group_id>`; a few legacy forms default to grouping method 0.

Group labels do not define chemical species. Add them only when a command or downstream tool needs a subset, region, molecule, or per-species group. Verify every referenced group contains the intended atoms.

## `run.in`

Each active line has:

```text
keyword parameter_1 parameter_2 ...
```

Blank lines and content after `#` are ignored. Commands execute in file order. A typical protocol is:

```text
replicate ...          # optional, before potential when required
potential ...
minimize ...           # optional immediate action
velocity ...
time_step ...

ensemble ...
dump_thermo ...
compute_... ...
dump_... ...
run ...

ensemble ...           # next stage
compute_... ...         # reissue non-propagating commands
run ...
```

Do not use this as a ready-made protocol. Load the relevant parameter references and ask for stage-specific values.

Command lifecycle matters:

- Immediate actions such as `potential`, `velocity`, `minimize`, and some static computes execute when parsed.
- `time_step` propagates across later `run` blocks.
- Ensembles, most computes, dumps, deformations, constraints, and driving fields are run-scoped or non-propagating.
- `run <number_of_steps>` executes the current stage and does not propagate.

Treat unfamiliar commands as run-scoped unless their bundled entry explicitly describes an immediate or propagating action. If parser behavior differs, stop and resolve the executable version instead of guessing.

## Auxiliary files

| File | Used by | Content |
|---|---|---|
| `kpoints.in` | `compute_phonon` | Fractional high-symmetry k-points and labels; blank line separates paths |
| `eigenvector.in` | GKMA/HNEMA/modal analysis | Harmonic eigenvectors generated by the documented modal workflow |
| Potential file | `potential` | Format-specific model parameters; preserve the supplied file and validate its header/species before use |
| `zbl.in` | Flexible NEP ZBL | Pair-ordered ZBL parameters when used by the model |
| PLUMED input | `plumed` | PLUMED configuration matching the compiled feature set |

## Input audit

- Confirm lattice determinant, cell orientation, PBC, and coordinate units.
- Confirm species and order match the potential file.
- Confirm every group method/id and charge/mass/velocity column.
- Confirm no atom rows are missing or have extra fields.
- Confirm every `run.in` keyword and signature exists in the bundled parameter catalogs.
- Confirm all filenames are resolved from the execution directory.
- Confirm append-mode outputs from earlier runs will not contaminate this run.
